# CLAUDE.md — проект dev-agent / продукт LogAnalyzer (la)

## Продукт

**LogAnalyzer (la)** — продукт для инженеров технической поддержки и аналитиков по анализу и визуализации лог-файлов заказчиков.

Состав продукта — **два независимо подключаемых и изменяемых проекта**:

1. **backend** (`release/backend/`) — поставщик API. Реализуется на **Go (первично)** и/или **Python/FastAPI (вторично, по запросу)**. Источник данных — **PostgreSQL**. Backend также раздаёт статический frontend (см. ниже).
2. **frontend** (`release/frontend/`) — клиент на **Angular**. Компилируется **build-time** Angular CLI (Node) в статический бандл `dist/`, который в runtime монтируется и раздаётся backend: Go — `embed.FS` + `http.FileServer`; Python/FastAPI — `StaticFiles`. **Node в runtime не используется.**

Проекты подключаются и развиваются независимо: изменения в одном не требуют перегенерации другого. В runtime работает один процесс backend (API + статика frontend).

## Назначение проекта dev-agent

dev-agent — проект разработки, в котором **Go первичен** как поставщик API (преимущественно JSON-данные), **Python (FastAPI) вторичен** и собирается только по отдельному запросу. Источник данных — **PostgreSQL**.

## Роль агента

Я — агент-разработчик и архитектор этого проекта. Веду проект от пользовательских историй к архитектуре и коду, поддерживаю соответствие кода и архитектурных артефактов, веду changelog, обеспечиваю изменяемость артефактов и кода без полной регенерации кодовой базы. Каждое действие проверяю компилятором и тестами; правки вношу по реальному выводу инструментов, а не наугад.

## Рабочий процесс (workflow)

Поток: **user-stories → architect → code → build**. Каждый шаг изменяем; код обязан соответствовать актуальной архитектуре.

1. `user-stories/` — пользовательские истории (`US-NNNN`), фиксируются **в двух форматах** (см. ниже).
2. `architect/` — архитектурные артефакты: спецификации, диаграммы, манифест трассировки.
3. `release/backend/go`, `release/backend/python/`, `release/frontend/` — сгенерированный код по спецификации.
4. `sh/` — скрипты управления процессом продукта (`start-la.sh`, `stop-la.sh`, `status-la.sh`).
5. `changelog/` — журнал изменений.

### Пользовательские истории — двойной формат

Каждая ЮС хранится в **двух форматах** одновременно:

- **Текстовый (человекочитаемый)** — `user-stories/US-NNNN-kebab-short-title.md` (Markdown с YAML-фронтматтером + тело с acceptance criteria). **Канонический источник истины для человека.**
- **Служебный (машинный)** — `user-stories/US-NNNN-kebab-short-title.json` (структурированное представление, удобное для детерминированной обработки: поля `id`, `title`, `status`, `priority`, `domain`, `target[]`, `spec_ref`, `asRole`, `wantGoal`, `soValue`, `context`, `acceptanceCriteria[]`, `nonFunctionalRequirements[]`, `dependencies[]`, `openQuestions[]`, `trace{}`). **Используется мной для обработки и трассировки.**

### Редактирование ЮС и распространение изменений

ЮС может быть поправлена **как через меня, так и внешним образом** через текстовый редактор (в дальнейшем — через git).

- Текстовый `.md` — канонический; при конфликте с `.json` приоритет у `.md`.
- Служебный `.json` я поддерживаю синхронным с `.md`; при обнаружении расхождения — сверяю, обновляю служебный формат и фиксирую дифф.
- **Любое изменение ЮС (внешнее или моё) подлежит проверке** и далее отражается в: архитектурных артефактах/схемах (`architect/`), коде (`release/`), тестах, документации, `changelog/CHANGELOG.md` и релизе продукта.
- Порядок обработки внешнего изменения: обнаружить дифф → проверить корректность (YAML/JSON валидны, нумерация/связи не нарушены) → обновить служебный `.json` → распространить по цепочке артефактов → записать в changelog → при необходимости пересобрать бинарник/релиз и обновить `architect/manifest.md`.

## Требования и ограничения

- **Продукт — LogAnalyzer (la):** анализ и визуализация лог-файлов заказчиков для тех.поддержки и аналитиков.
- **Два независимо подключаемых и изменяемых проекта:** backend (py/go) и frontend (angular).
- **Go первичен** — поставщик API (преимущественно JSON).
- **Python вторичен** (FastAPI) — собирается только по отдельному запросу; при этом по запросу должен быть доступен релиз на Python **в полном объёме функционала**.
- **Источник данных:** для MVP — **SQLite** (по умолчанию `SOURCE_DB_URL=sqlite:la.db`, авто-создание файла и схемы). **Postgres — позже**, отдельной ЮС; разбор scheme `postgres://` зарезервирован. Конфигурация БД и подключения читается из `la.conf` (создаётся из шаблона `la.conf.template` при отсутствии); дефолты — `SOURCE_DB_URL=sqlite:la.db`, `LISTEN_ADDRESS=localhost`, `LISTEN_PORT=8888`; путь к конфигу переопределяется env `LA_CONF`. Секреты в репозитории не хранятся; `la.conf` и `la.db` — runtime/сгенерированные (в `.gitignore`).
- **Управление процессом продукта** — через скрипты `sh/start-la.sh`, `sh/stop-la.sh`, `sh/status-la.sh` (старт, останов, проверка статуса). В runtime один процесс backend, раздающий API и статический frontend; Node в runtime не участвует.
- **Поставка (NFR):** дистрибутив для ПК пользователя — **только ZIP-архив**. Node.js используется **только на этапе сборки** frontend (Angular CLI); в поставку входит **собранный набор JavaScript** (Angular `dist/`); **Node.js не поставляется и не устанавливается** на ПК пользователя. Содержимое ZIP: бинарник backend (Go; Python — по запросу), собранный `dist/` frontend, `la.conf.template`, `sh/*.sh`.
- **Развёртывание (NFR):** после распаковки ZIP — при необходимости модифицируется/создаётся дефолтный `la.conf` (из шаблона), выполняется `sh/start-la.sh`, процесс стартует; при первом запуске создаётся дефолтная SQLite БД (`la.db`) и применяется схема (миграции); стартует веб-сервер со смонтированным Angular-приложением (backend раздаёт `dist/`). Внешних зависимостей на ПК пользователя нет (только ОС); инсталлятор отсутствует.
- **Парсеры — плагинная архитектура (NFR):** приложение определяет тип лога и применяет разные парсеры. Каждый парсер — отдельный подключаемый модуль: для Go-релиза — `.so` (Go plugin, `go build -buildmode=plugin`), для Python-релиза — `.py`; модули лежат в подкаталоге `parsers/` (путь переопределяется env `LA_PARSERS_DIR`). Добавление/модификация парсера: собирается/кладётся только модуль парсера в `parsers/`, затем выполняется **рестарт процесса** (рестарт допустим) — новый парсер подхватывается при старте сканированием `parsers/`. При этом **не требуется пересборка основного бинарника** и **не требуется повторное развёртывание всего продукта** (новый ZIP/дистрибутив не нужен). Модификация исходного кода парсера затрагивает только этот парсер. Общий стабильный интерфейс парсера — в отдельном пакете (Go: `internal/parser`; Python: `parsers/base.py`); изменение самого интерфейса — отдельная cross-cutting задача.
- **Ограничения Go-плагинов (учесть при реализации):** Go `plugin` — только Linux; плагин и хост собираются одним тулчейном (`/usr/local/go/bin/go`) и одинаковыми версиями общих зависимостей (гарантируется размещением исходников парсеров в том же модуле). Выгрузить загружённый `.so` в рамках работающего процесса нельзя, поэтому смена/добавление парсера выполняется через рестарт процесса (допустим по NFR) — без пересборки основного бинарника и без повторного развёртывания продукта.
- **Постобработчики — плагинная архитектура (NFR):** после загрузки и парсинга каждого файла выполняется этап постобработки — файл получает сводку. К каждому типу файла применяется свой постобработчик (параллельно парсерам): **базовый** постобработчик (built-in, всегда есть) показывает общие поля — число строк, дату загрузки, размер файла, кодировку, first/last ts, длительность, `level_counts`; **остальные постобработчики наследуют от базового** и расширяют информацию по событиям — как минимум: число сессий старт-стоп, длительности интервалов старт-стоп, число сообщений в категориях (error/warning/critical/etc). Постобработчики — отдельные подключаемые модули: для Go-релиза — `.so` (`go build -buildmode=plugin`), для Python-релиза — `.py`; лежат в подкаталоге `postprocessors/` (путь переопределяется env `LA_POSTPROCESSORS_DIR`). Добавление/модификация постобработчика: собирается/кладётся только модуль в `postprocessors/`, затем **рестарт процесса** (допустим) — подхватывается при старте сканированием `postprocessors/`; **без пересборки основного бинарника и без повторного развёртывания продукта**. Если плагина для формата нет — применяется built-in базовый постобработчик. Общий стабильный интерфейс постобработчика — в отдельном пакете (Go: `internal/postprocess`; Python: `postprocessors/base.py`); изменение интерфейса — отдельная cross-cutting задача. Сводка хранится в `t_files_analyze` (колонки `encoding`/`first_ts`/`last_ts`/`duration_sec` + `summary` JSON). Ограничения Go-плагинов — те же, что у парсеров.
- **Архитектура изменяема**; получаемый код должен соответствовать актуальной архитектуре.
- **Архитектурные артефакты (моя спецификация и пользовательский формат) должны совпадать и отражать код и бинарный образ.** Трассировка `спецификация → файлы → бинарник` хранится в `architect/manifest.md`.
- **Все сгенерированные файлы изменяемы по запросу** и развиваются без полной перегенерации всей кодовой базы (правки точечные, по конкретным артефактам).
- **Веду changelog при каждом изменении** функциональных и нефункциональных требований.
- **Git будет подключён позже**; пока работа идёт непосредственно в каталоге проекта.

## Правило нейминга файлов

- Пользовательские истории:
  - текстовый формат: `user-stories/US-NNNN-kebab-short-title.md`
  - служебный формат: `user-stories/US-NNNN-kebab-short-title.json` (тот же base-нейм, что у `.md`)
- Спецификации: `architect/specs/<domain>.spec.md`
- Диаграммы: `architect/diagrams/<domain>.<type>.mmd` (Mermaid) + встроенный рендер в `.spec.md`
- Go-файлы: `snake_case`; пакеты `lowercase`; форматирование через `gofmt`; точки входа в `release/backend/go/cmd/<name>/main.go`
- Python-файлы: `snake_case`; код в `release/backend/python/src/`, тесты в `release/backend/python/tests/`
- Frontend (Angular): conventional Angular-структура в `release/frontend/` (kebab-case для артефактов Angular)
- Скрипты управления процессом: `sh/start-la.sh`, `sh/stop-la.sh`, `sh/status-la.sh` (POSIX sh)
- Changelog: `changelog/CHANGELOG.md` (формат Keep a Changelog, SemVer); при необходимости per-release файлы `changelog/RELEASE-NNNN.md`
- Бинарники: backend Go — `release/backend/go/build/<name>`; backend Python — `release/backend/python/build/<name>`; frontend — `release/frontend/dist/` (сборка Angular)
