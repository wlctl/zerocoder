# Установка и развёртывание LogAnalyzer (la)

Поставка — **ZIP-архив без инсталлятора**. Никаких внешних зависимостей на ПК
пользователя нет (только ОС Linux). Node.js **не** требуется в runtime.

## 1. Распаковка

```sh
unzip la-<version>.zip
cd la-<version>
```

## 2. Конфигурация (опционально)

При первом старте backend создаёт `la.conf` из встроенного шаблона (если файла
нет). Шаблон `la.conf.template` приложен для справки. Поля по умолчанию:

| Поле                   | Значение по умолчанию                       | Назначение                          |
|------------------------|---------------------------------------------|-------------------------------------|
| `SOURCE_DB_URL`        | `sqlite:la.db`                              | БД (SQLite для MVP)                 |
| `LISTEN_ADDRESS`       | `localhost`                                 | адрес HTTP                          |
| `LISTEN_PORT`          | `8888`                                      | порт HTTP                           |
| `MAX_FILE_SIZE`        | `10GB`                                      | лимит входящего файла               |
| `MAX_FILE_COUNT`       | `10`                                        | число файлов за запрос (zip = 1)   |
| `LA_DEFAULT_TZ`        | `UTC`                                       | таймзона для логов без смещения     |
| `LA_PARSERS_DIR`       | `./parsers`                                 | каталог плагинов-парсеров           |
| `LA_POSTPROCESSORS_DIR`| `./postprocessors`                           | каталог плагинов-постобработчиков   |
| `LA_FRONTEND_DIST`     | `release/frontend/dist/la-frontend/browser` | каталог собранного Angular          |

**Приоритет:** переменные окружения перекрывают `la.conf` (`env > la.conf`).
Путь к конфигу — через `LA_CONF`.

В дистрибутиве стартовый скрипт `sh/start-la.sh` уже задаёт dist-относительные
пути (`LA_FRONTEND_DIST`, `LA_PARSERS_DIR`, `LA_POSTPROCESSORS_DIR`,
`SOURCE_DB_URL`) через env, поэтому правка `la.conf` для базового запуска не нужна.

## 3. Старт / статус / останов

```sh
sh/start-la.sh     # старт backend (раздаёт API + статический frontend)
sh/status-la.sh    # статус (RUNNING/STOPPED)
sh/stop-la.sh      # останов
```

При первом старте:
- создаётся `la.conf` (из шаблона);
- создаётся SQLite-БД `la.db` и применяется схема (миграции);
- загружаются плагины из `parsers/` и `postprocessors/` (сканирование `.so`);
- стартует веб-сервер на `LISTEN_ADDRESS:LISTEN_PORT`.

В runtime работает **один процесс** backend. Node.js не участвует.

## 4. Проверка

```sh
curl http://localhost:8888/healthz      # -> {"status":"ok"}
curl http://localhost:8888/api/parsers   # список загруженных парсеров/постобработчиков
```

Откройте `http://localhost:8888` — интерфейс Angular.

## 5. Перенос данных / миграции

Схема БД применяется идемпотентно при старте (таблица `schema_migrations`).
Удаление `la.db` сбрасывает все данные (для чистого старта).

## 6. Обновление дистрибутива

- **Бинарник/фронт:** замените `devagent` и `frontend/` — перезапустите.
- **Плагин:** замените/добавьте `.so` в `parsers/` или `postprocessors/` и
  перезапустите процесс — **без пересборки бинарника и без повторного
  развёртывания** (см. [docs/USAGE.md](USAGE.md) → «Плагины»).

## Примечания

- Go-плагины работают только на Linux; плагин и хост собираются одним тулчейном
  и одинаковыми версиями общих зависимостей (гарантируется одной поставкой).
- Выгрузить загруженный `.so` в работающем процессе нельзя — смена/добавление
  парсера/постобработчика выполняется через рестарт процесса (допустимо по NFR).
- `la.conf` и `la.db` — runtime/сгенерированные (не входят в ZIP).
