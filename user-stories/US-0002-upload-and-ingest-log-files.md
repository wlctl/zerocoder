# US-0002 — Загрузка и ингестия лог-файлов разнородных форматов

## Как инженер тех.поддержки/аналитик
## я хочу загрузить один или несколько лог-файлов (или zip-архив) через веб-интерфейс и автоматически разобрать их в таблицы для анализа
## чтобы повторно не возиться с сырыми логами и быстро перейти к поиску/визуализации.

## Контекст

Продолжение backend-потока (после US-0001 — конфиг и SQLite). Источник данных — SQLite (MVP).
Только текстовые лог-файлы. Логи разнородные: alert.log БД Oracle, лог БД Postgres, многострочные
Java-логи (weblogic/tomcat/glassfish), линуксовые лог-файлы. Загрузка через веб-интерфейс (Angular).
Ограничения и дефолтная таймзона — в конфиге: `MAX_FILE_SIZE=10GB`, `MAX_FILE_COUNT=10`, `LA_DEFAULT_TZ=UTC`.
Python-релиз полного функционала — по запросу (вторичный таргет).

## Решения по схеме/хранению (закрытые)

- **Хранение:** без BLOB и без отдельного ФС-хранилища — распарсенные записи (длинный TEXT) и есть хранилище.
- **Модель записей:** логическая запись = одна строка таблицы. Многострочный блок (datetime+критичность+начало сообщения, далее stack trace до следующей записи) склеивается в одну строку; `raw_line` (длинный TEXT) хранит полный текст блока с переносами; `ts`/`level` извлекаются из головной строки.
- **Datetime:** `ts` (UTC ISO8601, сортируемый), `ts_raw` (оригинал), `tz_offset` (смещение оригинала), `tz_inferred` (0/1 — смещение взято из `LA_DEFAULT_TZ`). Если дата не разобрана → `ts=NULL`, `ts_raw` сохранён.
- **Критичность:** `level` — нормализованное значение (error/warning/info/...).
- **Таблицы (миграция 0002):**
  - `t_files_upload` (вход): id, filename, md5 (UNIQUE — дедуп входящего), size_bytes, kind (zip|file), status, uploaded_at, error.
  - `t_files_analyze` (распакованные файлы, 1→N от upload): id, upload_id, filename, path_in_archive (подкаталог), md5, format, status, record_count, parsed_at, error.
  - `t_log_entries` (распарсенные записи): id, file_analyze_id, seq, format, ts, ts_raw, tz_offset, tz_inferred, level, component, message, raw_line (TEXT), attrs (JSON). INDEX (file_analyze_id, seq), (ts), (level).
- **zip:** считается за 1 в `MAX_FILE_COUNT`; размер входящего файла (архив или одиночный) ≤ `MAX_FILE_SIZE`; распаковка рекурсивна по подкаталогам (внутри только лог-файлы); архив не хранится; каждому распакованному файлу — новый ID в `t_files_analyze`.
- **Дедуп:** по MD5 **входящего** файла (`t_files_upload.md5`); повторная загрузка того же входящего файла блокируется сообщением «Файл уже был загружен ранее».
- **Формат:** автоопределение по имени/содержимому; не определился → `text` (построчный разбор + попытка вытянуть дату/уровень, `ts=NULL` если нет); файл всё равно просматриваем (в т.ч. текстовые конфиг-файлы). Форматы: `oracle`, `postgres`, `java`, `linux`, `text`.
- **Таймзона без смещения:** используется `LA_DEFAULT_TZ` (дефолт UTC, изменяемый), `tz_inferred=1`.

## Acceptance criteria

- [ ] AC-1: Веб-интерфейс позволяет выбрать и загрузить один или несколько лог-файлов либо один zip-архив.
- [ ] AC-2: Ограничения из конфига: `MAX_FILE_SIZE=10GB` (входящий файл), `MAX_FILE_COUNT=10` (zip считается за 1); при превышении — понятное сообщение.
- [ ] AC-3: zip распаковывается рекурсивно по подкаталогам; каждому файлу — новый ID; путь внутри архива сохраняется; архив не хранится.
- [ ] AC-4: Поддерживаются форматы `oracle` (alert.log), `postgres`, `java` (multiline weblogic/tomcat/glassfish), `linux`, плюс fallback `text`; формат определяется автоматически.
- [ ] AC-5: Повторная загрузка входящего файла с тем же MD5 блокируется сообщением «Файл уже был загружен ранее».
- [ ] AC-6: Пользователь может открыть список ранее загруженных файлов, повторно перейти к их анализу, либо удалить файл (каскадно с записями).
- [ ] AC-7: Загруженные файлы преобразуются в таблицу `t_log_entries` — одна логическая запись (многострочный блок целиком) = одна строка; `raw_line` — длинный TEXT (без BLOB).
- [ ] AC-8: Из каждой записи извлекаются datetime (с учётом форматов дат/таймзон; без смещения — `LA_DEFAULT_TZ`) и критичность; хранятся `ts` (UTC), `ts_raw`, `tz_offset`, `tz_inferred`, `level`.
- [ ] AC-9: Конфиг `la.conf`/`la.conf.template` дополнен полями `MAX_FILE_SIZE`, `MAX_FILE_COUNT`, `LA_DEFAULT_TZ` (дефолты 10GB/10/UTC).
- [ ] AC-10: Backend проверяется `go build/vet/gofmt/test -race/cover`; frontend — `ng build`/`ng test`.
- [ ] AC-11: После парсинга каждый файл получает сводку (постобработка): число записей, дата загрузки, размер, кодировка, first/last ts, длительность.
- [ ] AC-12: Форматные постобработчики (наследники базового) выделяют как минимум: число сессий старт-стоп, длительности интервалов старт-стоп, число сообщений в категориях (error/warning/critical/...).
- [ ] AC-13: Постобработчики — подключаемые модули (Go `.so` / Python `.py`) в подкаталоге `postprocessors/` (env `LA_POSTPROCESSORS_DIR`); добавление/модификация — только модуль + рестарт (без пересборки основного бинарника и без повторного развёртывания продукта); если плагина для формата нет — built-in базовый постобработчик.

## Non-functional requirements

- MD5 и парсинг — потоковые (без загрузки всего файла в память).
- Потоковая загрузка без полного буферирования в RAM.
- zip распаковывается потоком; защита от zip-бомб — ограничение по `MAX_FILE_SIZE` входящего.
- Нормализация дат: поддержка форматов каждого поддержанного типа логов + ISO8601/common patterns; unknown → `ts=NULL`.
- Секреты не в репозитории; конфиг-поля в `la.conf`/`la.conf.template`.
- Python-релиз полного функционала — по запросу.
- **Плагинная архитектура парсеров (NFR):** определение типа лога и применение разных парсеров; каждый парсер — отдельный модуль (Go: `.so` через `go build -buildmode=plugin`; Python: `.py`) в подкаталоге `parsers/` (путь — env `LA_PARSERS_DIR`). Добавление/модификация парсера: собирается/кладётся только модуль парсера в `parsers/`, затем **рестарт процесса** (допустим) — парсер подхватывается при старте сканированием `parsers/`; **без пересборки основного бинарника и без повторного развёртывания всего продукта**. Общий стабильный интерфейс — в отдельном пакете (Go: `internal/parser`; Python: `parsers/base.py`).
- **Ограничения Go-плагинов:** только Linux; плагин и хост — один тулчейн и одинаковые версии общих зависимостей (парсеры в том же модуле); выгрузка `.so` в работающем процессе невозможна — смена/добавление парсера через рестарт процесса (допустим по NFR), без пересборки основного бинарника и повторного развёртывания продукта.
- **Многоформатный/мультилокальный разбор дат (NFR):** datetime определяется и разбирается в разных форматах и локалях (ISO8601+offset, `dd/Mon/YYYY:HH:MM:SS +offset`, `YYYY-MM-DD HH:MM:SS,mmm`, `Mon DD, YYYY h:mm AM/PM TZ`, `Wed Mar 28 ... 2012` и др.); единый каталог layout-ов + локализованные имена месяцев (`Jan..Dec`, расширяемо); TZ-аббревиатуры неоднозначны → `LA_DEFAULT_TZ` (`tz_inferred=1`); ни один layout не подошёл → `ts=NULL`, `ts_raw` сохраняется. Общий модуль: Go `internal/parser/datetime.go`, Python `parsers/datetime.py`.
- **Этап постобработки (NFR):** после загрузки+парсинга каждый файл получает сводку. К каждому типу файла — свой постобработчик (параллельно парсерам): **базовый** постобработчик показывает число строк, дату загрузки, размер файла, кодировку и т.п.; **остальные наследуют от базового** и расширяют информацию по событиям — как минимум: число сессий старт-стоп, длительности интервалов старт-стоп, число сообщений в категориях (error/warning/critical/etc). Постобработчики — подключаемые модули (Go `.so` buildmode=plugin / Python `.py`) в подкаталоге `postprocessors/` (env `LA_POSTPROCESSORS_DIR`); добавление/модификация — только модуль + рестарт процесса (допустим), **без пересборки основного бинарника и без повторного развёртывания продукта**; если плагина для формата нет — built-in базовый постобработчик. Общий интерфейс — в отдельном пакете (Go `internal/postprocess`; Python `postprocessors/base.py`). Сводка хранится в `t_files_analyze` (колонки `encoding`/`first_ts`/`last_ts`/`duration_sec` + `summary` JSON) и доступна через `GET /api/files/{id}` и `POST /api/files/{id}/postprocess` (пере-запуск). Ограничения Go-плагинов — те же, что у парсеров.
