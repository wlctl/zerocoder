# US-0005 — Корреляция событий по времени между файлами

## Как аналитик тех.поддержки
## я хочу видеть события из нескольких выбранных файлов загрузки на едином таймлайне, выровненном по timestamp
## чтобы находить связанные события в разных лог-файлах (например, событие в серверном логе и ошибка в логе приложения в тот же момент).

## Контекст

Расширение домена `viewer` (поверх US-0003/US-0004). У аналитика часто один архив (zip) с несколькими лог-файлами или несколько загруженных файлов в одной загрузке (`t_files_upload` 1→N `t_files_analyze`). Существующие пофайловые таблицы (US-0003) показывают каждый файл отдельно; `search` (US-0004) ищет по тексту, но сортирует по `file_analyze_id, seq` — не выстраивает события разных файлов по времени. Нужно объединённое представление: записи из выбранных файлов, отсортированные по `ts` кросс-файл, с пометкой файла — чтобы видеть рядом события из разных логов, близкие по времени. Источник — SQLite (MVP). Go первичен; Python — полный функционал по запросу. Контракт — в `architect/specs/viewer.spec.md` (`correlate`).

## Что добавляется

- **Backend:** `GET /api/uploads/{id}/correlate?files=&from=&to=&q=&level=&limit=&offset=` — объединённый поток записей из указанных файлов (по умолчанию все файлы загрузки), отсортированный по `ts` кросс-файл (`ORDER BY ts IS NULL, ts, seq`), с `file_analyze_id` и `filename` (JOIN `t_files_analyze`); фильтры по таймлайн-окну/тексту/уровню; пагинация `{items, total, limit, offset}`. Записи без `ts` (NULL) — в конце. Лимиты как у `search` (limit ≤ 1000).
- **Frontend:** режим «Корреляция (общий таймлайн)» в просмотрщике — переключатель заменяет пофайловые таблицы единой объединённой таблицей: колонки `ts | файл (цветовая метка) | level | component | message | raw`; строки из `/correlate` по выбранным в селекторе файлам + таймлайн-окно + активный поиск; подкраска (highlights) применяется; пагинация. Над таблицей — общий график событий по выбранным файлам (`/histogram?files=...`, группировка месяц/день/час/минута). Цвет файла — стабильная палитра по индексу.
- Переиспользуются: `timeline` (границы), `histogram` (мульти-файловый, уже есть `files[]`), `highlights` (подкраска), селектор файлов (US-0003).

## Non-functional requirements

- Объединённый поток — серверная сортировка по `ts` (индекс `idx_log_entries_ts`); JOIN за имя файла; лимит записей (защита от тяжёлых выборок).
- Записи без ts не теряются (в конце потока), но на общий таймлайн-график не попадают (histogram уже фильтрует `ts IS NOT NULL`).
- Секреты не в репозитории.
- Python-релиз — тот же эндпоинт/контракт (полный функционал по запросу).
- Go-плагины (парсеры/постобработчики) — не затрагиваются.

## Зависимости

- US-0002 (таблицы `t_files_analyze`/`t_log_entries`), US-0003 (селектор файлов, подкраска, график), US-0004 (`timeline`, `histogram?files=`, `search`-фильтры) — расширяются/переиспользуются.

## Открытые вопросы (уровень спецификации)

- Стекированный по файлам график (вместо суммарного) — на потом (MVP: суммарный `histogram?files=...`).
- FTS5/регулярки для `q` — отдельная ЮС (MVP: LIKE).
- Цветовая палитра файлов — детерминированная по индексу (расширяемо).
